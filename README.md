# Keyboard Protector PoC

`keyprotect-poc`는 Windows 전역 키보드 훅(`WH_KEYBOARD_LL`)과 프로세스별 허용 목록을 이용해 입력을 가로채고, 허용된 프로세스에만 `SendInput`으로 다시 전달하는 실험용 프로젝트입니다.

이 저장소의 현재 목적은 "키 입력을 안전하게 암호화하는 보안 제품"을 만드는 것이 아니라, "프로세스 기반 입력 차단/재주입 방식이 어떻게 동작하는지 확인하는 데모"를 제시하는 데 있습니다.

## 목표

이 프로젝트가 지금 해결하려는 문제는 다음입니다.

- Windows 저수준 키보드 훅으로 키 입력을 가로채는 기본 흐름을 실험한다.
- 활성 창의 프로세스 이름을 기준으로 입력 허용/차단 정책을 적용한다.
- 허용된 프로세스에 대해서만 입력을 다시 주입하는 구조를 검증한다.
- 전역 입력 제어 기능을 구현할 때 필요한 안정성 이슈를 드러내는 PoC로 사용한다.

## 비목표

현재 구현은 다음을 해결하지 않습니다.

- 키 입력의 기밀성을 보장하는 실제 암호화
- 키로거, 커널 모드 후킹, 관리자 권한 악성코드에 대한 방어
- 민감 정보 보호 제품 수준의 보안 경계 제공
- IME, 조합키, 스캔코드, 레이아웃 차이까지 보존하는 완전한 입력 전달
- 실사용 환경에 바로 배포 가능한 안전한 차단 도구

특히 `encrypt_keycode_with_salt()`와 `decrypt_keycode_with_salt()`는 XOR 기반 샘플일 뿐이며, 보안 기능으로 간주하면 안 됩니다.

## 위협 모델

현재 README 기준으로 정의하는 현실적인 위협 모델은 아래와 같습니다.

- 보호하려는 대상:
  - 사용자가 활성화한 특정 프로세스에만 키 입력이 전달되도록 제한하는 실험
  - 전역 훅 기반 입력 차단/전달 구조의 동작 확인
- 보호하지 못하는 대상:
  - 같은 시스템에서 더 높은 권한을 가진 프로세스
  - 커널 드라이버, 디버거, 다른 후킹 도구
  - 메모리 내 평문 키 값, 콘솔 로그, 재주입 이벤트 관찰자
- 신뢰 경계:
  - 훅 설치, 프로세스 판별, XOR 처리, `SendInput` 재주입이 모두 같은 프로세스 안에서 수행된다.
  - 따라서 "가로채서 암호화한 뒤 안전하게 전달한다"는 보안 경계는 현재 존재하지 않는다.

## 왜 이 프로젝트가 필요한가

이 코드는 보안 제품의 완성형이라기보다, 다음 질문에 답하기 위한 출발점으로 의미가 있습니다.

- 전역 키보드 훅으로 모든 입력을 막으면 어떤 운영 위험이 생기는가?
- 허용 프로세스 기반 재주입 구조에서 재진입, 중복 입력, 키업/키다운 불일치가 어떻게 나타나는가?
- 입력 보호를 주장하려면 문서, 안전장치, 설계 경계가 얼마나 더 필요해지는가?

즉, 이 프로젝트의 현재 가치는 "보호 기능 제공"보다 "문제와 한계를 빠르게 드러내는 실험 환경"에 가깝습니다.

## 현재 동작 방식

프로그램은 대략 다음 순서로 동작합니다.

1. `SetWindowsHookEx(WH_KEYBOARD_LL, ...)`로 전역 저수준 키보드 훅을 설치합니다.
2. 실행 파일 기준 경로의 `config.ini`에서 허용 프로세스 목록을 읽습니다.
3. 키다운 시 활성 창의 프로세스 이름을 확인합니다.
4. 키코드를 XOR 기반 함수로 변환한 뒤 즉시 다시 복호화합니다.
5. 허용된 프로세스면 `SendInput`으로 키를 재주입하고, 원래 입력은 차단합니다.
6. 허용되지 않은 프로세스면 입력을 그대로 차단합니다.
7. `Esc`를 누르면 메시지 루프를 종료하고 훅을 해제합니다.

## 현재 상태 요약

- 전역 입력을 가로채는 데모는 동작합니다.
- 허용 목록은 `config.ini`의 `[AllowedProcesses]` 섹션에서 읽습니다.
- 허용 목록에 없는 프로세스의 입력은 모두 차단됩니다.
- 설정 파일이 없거나 비어 있으면 `notepad++.exe`만 기본 허용됩니다.
- 디버그성 로그가 매우 많고, 키코드와 salt 값도 출력됩니다.
- 재주입 이벤트 식별, 안전 모드, 우회 키, 타임아웃 해제 같은 보호 장치는 아직 없습니다.

## 알려진 제한과 위험

이 프로그램은 실제 사용 시 시스템 사용성을 크게 해칠 수 있습니다.

- 허용되지 않은 프로세스에서는 모든 입력이 차단됩니다.
- 종료 키가 사실상 `Esc` 하나뿐이라 상황에 따라 복구가 어려울 수 있습니다.
- 훅 콜백 안에서 프로세스 조회, 로그 출력, XOR 처리, `SendInput` 호출이 모두 수행됩니다.
- `SendInput`으로 보낸 이벤트를 다시 훅이 잡을 수 있어 재진입 문제가 생길 수 있습니다.
- 문자 입력 의미 보존이 보장되지 않으므로 IME, 조합키, 확장키에서 오동작할 수 있습니다.
- 민감한 키 입력 정보가 콘솔 로그로 노출될 수 있습니다.

실사용 장비보다는 테스트 VM이나 복구 가능한 환경에서만 다루는 편이 안전합니다.

## 빌드

요구 사항:

- Windows
- MinGW-w64 또는 GCC 호환 환경
- `make`
- 링크 라이브러리: `user32`, `advapi32`, `psapi`, `shlwapi`

기본 빌드:

```cmd
make
```

자주 쓰는 타깃:

```cmd
make clean
make rebuild
make debug
make release
make run
make help
```

## 설정 파일

`config.ini` 예시는 다음과 같습니다.

```ini
[AllowedProcesses]
Process1=notepad++.exe
Process2=notepad.exe
Process3=code.exe
```

주의할 점:

- 프로세스 이름 비교는 대소문자를 구분하지 않습니다.
- 현재 구현은 실행 파일 이름만 비교합니다.
- 잘못된 설정 검증, 중복 제거, 최대 개수 초과 처리 강화는 아직 되어 있지 않습니다.
- 설정 파일이 없으면 `notepad++.exe` 하나만 허용하는 기본값으로 돌아갑니다.

## 실행

```cmd
bin\keyboard_protector.exe
```

실행 후:

- 허용된 프로세스에서는 키 입력이 재주입됩니다.
- 허용되지 않은 프로세스에서는 키 입력이 차단됩니다.
- `Esc`를 누르면 종료를 시도합니다.

강제 종료가 필요하면 다음 명령을 사용할 수 있습니다.

```cmd
taskkill /F /IM keyboard_protector.exe
```

## 라이선스와 사용 주의

이 프로젝트는 연구와 교육용 PoC로 다루어야 합니다. 운영 환경이나 민감 입력 보호 제품처럼 소개하거나 배포하기에는 아직 안전장치와 검증이 부족합니다.
