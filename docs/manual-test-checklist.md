# 수동 테스트 체크리스트

이 문서는 `keyprotect-poc`의 반복 가능한 수동 검증 절차를 정의한다.

## 테스트 환경

- OS: Windows 10/11 (권장: 테스트 VM)
- 빌드: `make debug` 또는 `make release`
- 대상 실행 파일: `bin\keyboard_protector.exe`
- 로그 확인 파일: 실행 경로의 `keylog.txt`
- 기본 설정 파일: 실행 경로의 `config.ini`

## 결과 기록 템플릿

각 테스트 케이스마다 아래 항목을 남긴다.

- 실행 날짜/시간
- 테스트 담당자
- 실행 권한 (관리자/일반)
- 테스트 ID
- 기대 결과
- 실제 결과
- Pass/Fail
- 비고(재현 절차, 스크린샷 경로, 로그 발췌)

## 체크리스트

### TC-01 허용 프로세스 입력 재주입

1. `config.ini`의 `[AllowedProcesses]`에 `notepad.exe`를 등록한다.
2. 보호기 실행 후 Notepad를 활성 창으로 둔다.
3. 알파벳, 숫자, Backspace를 입력한다.
4. `keylog.txt`에 `Allowed=yes`가 기록되는지 확인한다.

기대 결과:
- 키 입력이 Notepad에 정상 반영된다.
- 동일 키가 중복 입력되지 않는다.

### TC-02 비허용 프로세스 입력 차단

1. 허용 목록에 없는 앱(예: 계산기)을 활성 창으로 둔다.
2. 여러 키를 입력한다.
3. `keylog.txt`에서 `Allowed=no` 기록을 확인한다.

기대 결과:
- 해당 앱에서 입력이 차단된다.
- 프로세스 전환 후 허용 앱에서는 다시 입력된다.

### TC-03 종료 동작 (`Esc`)

1. 보호기 실행 상태에서 `Esc`를 누른다.
2. 콘솔 종료 메시지와 상태 코드를 확인한다.

기대 결과:
- 메시지 루프가 정상 종료된다.
- 훅 해제 후 프로세스가 종료된다.

### TC-04 비상 우회 토글 (Ctrl+Alt+Shift+Pause)

1. 보호기 실행 후 우회 토글 키를 누른다.
2. 비허용 앱에서 키 입력을 수행한다.
3. 다시 토글 키를 눌러 우회를 해제한다.

기대 결과:
- 우회 활성화 중에는 차단이 일시 해제된다.
- 우회 해제 후 정책이 다시 적용된다.

### TC-05 잘못된 `config.ini` 처리

1. 설정 파일을 제거하거나, 섹션/값을 비워둔 상태로 실행한다.
2. 콘솔 경고 메시지와 degraded status를 확인한다.

기대 결과:
- 프로그램이 치명 오류 없이 실행된다.
- fail-open으로 차단이 비활성화된다.

### TC-06 관리자/비관리자 실행 비교

1. 일반 권한으로 실행 후 동작을 확인한다.
2. 관리자 권한으로 실행 후 같은 시나리오를 반복한다.

기대 결과:
- 훅 설치 실패/경고가 권한 상태에 맞게 출력된다.
- 관리자 여부 로그가 기대대로 표시된다.

### TC-07 IME/조합키 확인

1. 한글 IME 활성 상태에서 자모 조합 입력을 수행한다.
2. `Shift`, `Ctrl`, `Alt` 조합키와 숫자패드 키를 입력한다.

기대 결과:
- 현재 구현 한계(의미 보존 실패 가능성)를 식별하고 기록한다.
- 재현 가능한 오동작 케이스를 남긴다.

## 실행 권장 순서

1. TC-05 (안전 모드/설정 이상)
2. TC-01 ~ TC-04 (핵심 동작)
3. TC-06 (권한 차이)
4. TC-07 (입력 의미 보존 한계)

## 판정 기준

- Critical: 입력 장치가 복구되지 않거나 프로세스 종료 불가
- Major: 허용/차단 정책이 역전되거나 중복 입력 발생
- Minor: 로그 누락, 메시지 표현 불일치, 경미한 UI/출력 문제
