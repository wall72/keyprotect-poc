# WORKLIST

## 목적

현재 `keyprotect-poc`는 "동작 데모" 수준의 뼈대는 갖추고 있지만, 실제 사용/확장/검증을 하기에는 보완할 부분이 많다.  
이 문서는 현 소스 기준의 불완전한 지점을 정리하고, 우선순위와 완료 상태가 함께 보이는 실행용 작업 문서다.

## 현재 상태 요약

- 전역 키보드 훅(`WH_KEYBOARD_LL`)을 설치해 키 입력을 가로챈다.
- 활성 창의 프로세스명이 허용 목록에 있으면 `SendInput`으로 키를 다시 전달한다.
- 허용되지 않은 프로세스의 입력은 모두 차단한다.
- 키코드 "암호화"는 XOR 기반 샘플 구현이며 보안 기능이라고 보기 어렵다.
- README는 목표/비목표/위협 모델 중심으로 정리되었지만, 코드 주석과 일부 메시지는 아직 정리가 필요하다.
- 테스트 코드와 안전장치가 거의 없어 실제 사용 시 오동작 위험이 크다.

## 핵심 진단

### 1. 현재 구현은 보안 제품이라기보다 입력 차단 데모에 가깝다

- `encrypt_keycode_with_salt()` / `decrypt_keycode_with_salt()`는 단순 XOR이라 보호 효과가 거의 없다.
- 복호화한 값을 같은 프로세스 안에서 즉시 다시 `SendInput` 하는 구조라 "보호" 경계가 없다.
- 민감 입력 보호 목적이라면 설계 자체를 다시 정의해야 한다.

### 2. 훅 콜백에서 너무 많은 일을 하고 있다

- 프로세스 조회, 로그 출력, 설정 판단, 암복호화, `SendInput` 호출이 훅 콜백 안에 몰려 있다.
- 저수준 키보드 훅은 매우 빈번하게 호출되므로 지연, 누락, 예기치 않은 재진입 문제를 만들기 쉽다.

### 3. 운영 안정성과 사용자 안전장치가 부족하다

- 허용되지 않은 프로세스는 전부 입력 차단이라 사용자가 쉽게 자기 시스템을 잠글 수 있다.
- 종료 키가 `ESC` 하나뿐이라 특정 상황에서는 정상 종료가 어려울 수 있다.
- 로그가 콘솔에 과도하게 출력되어 성능과 보안 양쪽에서 부담이 있다.

### 4. 문서/빌드/설정 품질이 낮다

- README는 정리되었지만 코드 주석과 콘솔 메시지에 인코딩 깨짐이 남아 있다.
- 빌드 타깃과 실제 소스 상태를 검증하는 절차가 없다.
- 설정 스키마가 단순하고 예외 처리/검증이 부족하다.

- 최소한 다음 중 하나는 필요:
  - 안전 모드
  - 임시 bypass 키
  - 타임아웃 자동 해제
  - 관리자 콘솔 종료 명령
  - 화이트리스트 비어 있을 때 차단 비활성화

완료 기준:

- [ ] 잘못된 설정 파일이나 예외 상황에서도 사용자가 복구 가능

#### [ ] 4. 훅 콜백 경량화

- 훅 내부에서 무거운 작업 최소화

- 필요하면 이벤트 큐/버퍼로 넘기고 별도 처리 스레드에서 로깅

완료 기준:

- [ ] 훅 콜백은 "판단 + 최소 전달"만 수행
