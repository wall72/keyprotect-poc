# WORKLIST

## 목적

`keyprotect-poc`는 Windows 전역 키보드 훅과 프로세스별 허용 목록을 이용해 입력 차단/재주입 구조를 실험하는 PoC다.
현재 데모는 동작하지만, 실사용 관점의 안전장치, 설정 검증, 종료 복구 흐름, 문서 정합성은 아직 부족하다.
이 문서는 구현의 한계와 우선순위를 정리하고, 완료 여부가 바로 보이도록 추적하기 위한 작업 목록이다.

## 현재 상태 요약

- 전역 저수준 키보드 훅(`WH_KEYBOARD_LL`)을 설치해 키 입력을 가로챈다.
- 활성 창의 프로세스 이름이 허용 목록에 있으면 `SendInput`으로 다시 전달한다.
- 허용되지 않은 프로세스의 입력은 차단한다.
- XOR 기반 키코드 변환은 샘플 수준이며 보안 기능으로 볼 수 없다.
- README는 현재 목적과 한계를 반영하도록 정리되었다.
- 설정 검증, 종료 흐름, 상태 코드 체계는 1차 정리되었지만 추가 안전장치는 더 필요하다.
- 자동 테스트는 없고, 수동 검증 의존도가 높다.

## 문제 진단

### 1. 현재 구현은 보안 제품보다 입력 차단 데모에 가깝다

- `encrypt_keycode_with_salt()` / `decrypt_keycode_with_salt()`는 단순 XOR 샘플이라 보호 효과가 거의 없다.
- 같은 프로세스 안에서 가로채고 다시 `SendInput`으로 재주입하므로 보안 경계가 성립하지 않는다.
- README와 코드, 로그, 함수명 사이의 표현도 아직 완전히 정렬되지는 않았다.

### 2. 훅 콜백에 책임이 많이 몰려 있다

- 프로세스 조회, 허용 판단, 로그 기록, 키 상태 추적, 재주입이 한 파일과 한 흐름에 집중되어 있다.
- 저수준 훅은 매우 자주 호출되므로 작은 비효율도 사용성 문제로 이어질 수 있다.

### 3. 운영 안전장치가 충분하지 않다

- 허용되지 않은 프로세스에서는 모든 입력이 차단된다.
- 종료는 기본적으로 `Esc` 의존이 크고, 비정상 상황에서 복구 수단이 넉넉하지 않다.
- 차단을 비활성화하는 fail-open 동작은 추가되었지만, 더 명시적인 안전 모드와 우회 수단이 필요하다.

### 4. 문서와 검증 자산이 더 필요하다

- README는 정리됐지만 코드 주석, 로그 표현, WORKLIST, 빌드 스크립트 설명은 추가 정리가 필요하다.
- 실제 동작 범위와 한계를 반복 가능하게 확인할 테스트 체크리스트가 없다.

## 우선순위별 작업

### Priority 1. 안전장치와 동작 경계 정리

#### [x] 1. 프로젝트 목표 재정의

- README에 목표, 비목표, 위협 모델, 사용 주의를 반영했다.
- 현재 구현을 “보안 제품”이 아니라 “프로세스 기반 입력 차단/재주입 PoC”로 명확히 표현했다.

완료 기준:

- [x] README만 읽어도 현재 목적과 한계를 이해할 수 있다.
- [x] 보안 경계가 없다는 점이 문서에 분명히 드러난다.

#### [x] 2. 재주입 이벤트 재진입 방지

- `SendInput`으로 보낸 이벤트가 다시 훅에 잡혀 무한 루프에 빠지지 않도록 식별 태그를 추가했다.
- 자기 자신이 재주입한 이벤트를 구분해 중복 전달을 막았다.

완료 기준:

- [x] 허용된 프로세스에서 키다운/키업이 1회씩만 전달된다.
- [x] 자기 재주입 이벤트로 인한 반복 호출이 발생하지 않는다.

#### [ ] 3. 전체 차단 안전장치 추가

- 전역 입력 차단 기능은 기본적으로 위험하다.
- 최소한 아래 중 일부 또는 전부가 더 필요하다.
- 안전 모드
- 임시 bypass 키
- 타임아웃 기반 자동 해제
- 관리자 콘솔 종료 명령
- allowlist가 비어 있을 때 차단 비활성화

완료 기준:

- [ ] 잘못된 설정이나 예외 상황에서도 사용자가 복구 가능한 경로가 있다.

#### [x] 4. 훅 콜백 경량화

- 디버그 로그를 줄이고 중복 처리 경로를 단순화했다.
- 최소한의 상태 추적만 남기고, 기본 흐름을 읽기 쉽게 정리했다.

완료 기준:

- [x] 훅 콜백은 차단/허용 판단과 최소 전달에 집중한다.
- [x] 기본 빌드에서 과도한 콘솔 I/O가 없다.

### Priority 2. 설정, 종료, 오류 흐름 정리

#### [ ] 5. 입력 처리 로직 모듈 분리

- 현재 `keyboard_protector.c`에 책임이 많이 몰려 있다.
- 다음과 같은 분리를 검토한다.
- `hook.c`: 훅 설치/해제, 콜백
- `process_filter.c`: 활성 프로세스 조회, 허용 판단
- `config.c`: INI 로드와 검증
- `input_forwarder.c`: `SendInput` 래핑
- `logger.c`: 로그 레벨과 출력 관리

완료 기준:

- [ ] 파일별 책임이 명확하고 테스트 가능한 함수 단위가 드러난다.

#### [x] 6. 설정 파일 검증 강화

- 중복 프로세스명을 제거한다.
- 공백과 따옴표를 정리한다.
- 경로 입력은 실행 파일명 기준으로 정규화한다.
- 최대 개수 초과는 경고 후 무시한다.
- 설정이 없거나 유효한 항목이 없으면 차단을 비활성화한 상태로 계속 실행한다.

완료 기준:

- [x] `config.ini`가 불완전해도 복구 가능한 동작을 유지한다.
- [x] 잘못된 항목은 경고 후 무시되고 유효한 항목만 반영된다.

#### [x] 7. 종료 흐름 개선

- 종료 요청 경로를 한 함수로 모은다.
- 메시지 루프 종료, 훅 해제, 상태 정리, 로그 종료 순서를 일관되게 유지한다.
- 초기화 실패와 런타임 종료를 같은 기준으로 정리한다.

완료 기준:

- [x] 종료 경로가 한 흐름으로 정리되어 있다.
- [x] 비정상 종료 시에도 정리 루틴이 빠지지 않는다.

#### [x] 8. 상태/오류 코드 체계화

- `printf` / `fprintf` 중심의 단편적 오류 표시를 보완한다.
- 초기화, 설정, 로그, 메시지 루프 오류를 구분할 상태 코드를 둔다.

#### [ ] 9. “보호” 표현 재검토 또는 구현 교체

- 실제 보호가 목적이면 다른 설계와 보안 API 검토가 필요하다.

완료 기준:
